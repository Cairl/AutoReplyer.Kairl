## AutoReplyer.Kairl — 修复 CuPy 模式导致回复失败

### 根因
`_find_lowest_bubble()` 包含大量 Python 控制流（while/if/标量索引），当传入 cupy 数组时，GPU 驻留数组在 Python 布尔上下文中行为不确定，导致异常被 `_scan_group` 的 `except Exception` 静默吞掉。

### 修复
将 `_analyze_region()` 重构为 "GPU 计算 + CPU 控制流" 混合模式：
- 颜色匹配（`xp.all`/`xp.abs`）在 GPU 上完成后，立即 `xp.asnumpy()` 回 CPU
- `_find_lowest_bubble()` 始终接收 numpy 数组（函数签名增加 NOTE 说明）
- `_find_lowest_bubble()` 内部恢复使用 `numpy.where()` 替代 `xp.where()`
- 对检测（row_std）同样先 `xp.std()` 在 GPU，再 `xp.asnumpy()` 回 CPU 后用 `numpy.where()`
- `asnumpy()` 调用用 `hasattr(xp, "asnumpy")` 守卫，确保 numpy 路径不受影响（numpy 无此属性）

### 修改文件
- `core/monitor.py`：`_analyze_region()` + `_find_lowest_bubble()`
- `core/gpu.py`：CuPy 缺失时改为硬 ImportError（强制安装）
- `ui/tui.py`：文本 "GPU加速" → "硬件加速"，4 态简化为 3 态

## AutoReplyer.Kairl — 修复夹杂消息导致错误回复 + overlay 合并框

### 问题
用户反馈两个问题：
1. 视觉框选只框接收消息，配对时（接收+回复）应合并为一个整体框
2. 中间夹杂其他消息时会错误回复，例如：@所有人 → 我:2 → 测试 → 测试 → 我:2（最后一条 2 是错误回复）

### 根因
`_analyze_region` 用"接收气泡下方右半部分的像素方差(std>30)"判断是否已回复：
- 不区分内容类型：任何内容（非绿色回复）都会导致高方差→误判已回复
- 几何脆弱：回复气泡若在接收气泡右侧（同高）而非下方，方差判断区域漏掉它→误判未回复→错误触发
- 只检查最底部接收气泡，配对关系不明确

### 修复（core/monitor.py + core/overlay.py）
1. **精确识别绿色回复气泡**：新增 `_LIGHT_REPLY`/`_DARK_REPLY` 颜色 mask（#9DF29F/#35D28D），用颜色匹配代替泛泛的方差判断
2. **新增 `_find_all_bubbles`**：返回所有气泡 bbox（不只是最底部），用于建立配对关系
3. **贪心配对算法**：每个接收气泡（从上到下）配对它之后、下一个接收气泡之前最近的未用绿色回复。正确处理夹杂消息——回复归上方消息，不被下方消息偷走
4. **overlay 配对合并框**：`show(region, reply_region, paired=True)` 时画一个青色 #8CDEF6 合并框（接收+回复的 bounding box 并集），表示"这对上下文已整体处理"
5. **顺带修复 overlay bug**：原 `_drain_queue` 只传 `msg["region"]` 给 `_draw`，`reply_region` 在 msg 顶层从未传递→绿框实际从未显示。改为传整个 msg

### 关键设计
- 配对容差：`rp[1] >= rb[1] - 2`（回复顶部≥接收顶部-2px），覆盖回复在接收右侧同高的情况
- 回复按 y 排序后用 break 提前退出（reply 属于下一个接收气泡时，后续都属下一个）
- 最底部接收气泡的配对状态决定 `already_replied`；触发条件不变（OCR含关键词 + 无配对回复 + 非in-flight）

### 测试（tests/test_pairing.py，7场景全过）
1. 夹杂消息：@all→2→测试→测试→2，最底部"测试"配对最后的2，已回复
2. 单个@all+回复：配对，已回复
3. @all无回复：未回复（会触发）
4. 回复不被下方接收消息偷走：R1归A，B未配对
5. 上方已回复下方未回复：下方正确触发
6. 日间模式颜色 #EEEEF0
7. 暗色绿色回复 #35D28D

### 修改文件
- `core/monitor.py`：`_analyze_region` 重写 + 新增 `_find_all_bubbles` + `_scan_group` 两处 overlay 调用传 paired
- `core/overlay.py`：`show` 新增 paired 参数 + `_drain_queue`/`_draw` 重写支持合并框
- `tests/test_pairing.py`（新建）：配对逻辑单元测试

## AutoReplyer.Kairl — 修复 overlay 框逐渐缩小 + 无关消息被青框框选

### 问题1：框逐渐缩小直到失效再恢复
**根因**：overlay 窗口对 pyautogui.screenshot 可见。框轮廓（2-3px）压在气泡边缘像素上，下次截图拍到框颜色而非气泡色→检测到的气泡 bbox 变小→画更小的框→再吃边缘→循环缩小直到检测失败框消失→下次又检测到完整气泡→周而复始。

**修复（core/overlay.py）**：
1. 新增 `_exclude_from_capture(root)`：用 `SetWindowDisplayAffinity(WDA_EXCLUDEFROMCAPTURE=0x11)` 让 overlay 窗口对用户可见但对所有屏幕截图（BitBlt/PrintWindow/DirectX）完全隐身。要求 Win10 2004+（本机 Win11 26200，实测返回 True）
2. 64位安全：`SetWindowDisplayAffinity.argtypes = [c_void_p, c_uint]`，避免 HWND 被截断为 32 位 int
3. HWND 获取：`winfo_id()` 是 tkinter 子窗口，`GetParent()` 取真实顶层
4. 兜底：capture exclusion 失败时（老系统），`_draw` 给所有框加 3px 外扩 margin，让轮廓落在气泡外侧而非边缘像素上

### 问题2：无关消息也被青色框框选
**根因**：`_scan_group` 在 OCR 关键词匹配**之前**就调 `overlay.show()`。所以即使接收消息不含触发词（如"其他人无关话+我无关回复"配对），already_replied=True 也会画青色合并框。

**修复（core/monitor.py）**：重构 `_scan_group` 控制流——把 `overlay.show()` 全部移到关键词匹配之后：
1. 截图→分析→OCR→关键词检查（此段无 overlay）
2. 非触发词 → `return`（不画任何框）
3. 触发词 → 首次 `overlay.show(paired=already_replied)`；若已配对则清 inflight 并 return
4. 点击暴露→重新截图分析→第二次 `overlay.show`
5. 新增 `_overlay_view(region, view)` 辅助方法消除两处 overlay 调用的重复代码

### 验证
- 语法检查通过（py_compile）
- SetWindowDisplayAffinity 实测返回 True（本机 Win11 26200）
- 配对测试 7/7 仍全过（重构未破坏 _analyze_region）
- 仅 2 处 `_overlay.show` 调用，均用新 `paired=` 签名

### 修改文件
- `core/overlay.py`：新增 `_exclude_from_capture` + `_build_window` 调用它 + `_draw` 加 margin 兜底 + 模块 docstring 补 capture exclusion 说明
- `core/monitor.py`：`_scan_group` 重构（overlay 后移）+ 新增 `_overlay_view` 辅助方法
