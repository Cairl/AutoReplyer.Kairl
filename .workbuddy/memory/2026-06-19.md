## AutoReplyer.Kairl — 修复 CuPy 模式导致回复失败

### 根因
`_find_lowest_bubble()` 包含大量 Python 控制流（while/if/标量索引），当传入 cupy 数组时，GPU 驻留数组在 Python 布尔上下文中行为不确定，导致异常被 `_scan_group` 的 `except Exception` 静默吞掉。

### 修复
将 `_analyze_region()` 重构为 "GPU 计算 + CPU 控制流" 混合模式：
- 颜色匹配（`xp.all`/`xp.abs`）在 GPU 上完成后，立即 `xp.asnumpy()` 回 CPU
- `_find_lowest_bubble()` 始终接收 numpy 数组（函数签名增加 NOTE 说明）
- `_find_lowest_bubble()` 内部恢复使用 `numpy.where()` 替代 `xp.where()`
- 对检测（row_std）同样先 `xp.std()` 在 GPU，再 `xp.asnumpy()` 回 CPU 后用 `numpy.where()`
- `asnumpy()` 调用用 `hasattr(xp, "asnumpy")` 守卫，确保 numpy 路径不受影响（numpy 无此属性）

### 修改文件
- `core/monitor.py`：`_analyze_region()` + `_find_lowest_bubble()`
- `core/gpu.py`：CuPy 缺失时改为硬 ImportError（强制安装）
- `ui/tui.py`：文本 "GPU加速" → "硬件加速"，4 态简化为 3 态

## AutoReplyer.Kairl — 让 RTX 5070 GPU 加速真正生效

### 背景
上一轮修复后 TUI 显示"已回退 CPU (CuPy 运行时不可用: RuntimeError)"。用户有 RTX 5070，明确要求用 GPU 加速。

### 根因
- cupy-cuda12x 14.1.1 已装，但缺 CUDA toolkit headers（nvidia wheels 没装）
- RTX 5070 是 Blackwell (sm_120)，需要 CUDA 12.8+
- cupy._environment 探测系统级 CUDA 失败 → JIT 编译 kernel 抛 RuntimeError

### 解决
1. `pip install "cupy-cuda12x[ctk]"` 安装 CUDA 12.9.2 全套 wheels（nvrtc 12.9.86 等，支持 sm_120）
2. core/gpu.py：import cupy 前 filterwarnings 过滤 "CUDA path could not be detected" 假阳性警告（wheel 模式下 cuda-pathfinder 自动找库，cupy._environment 的系统级探测是过时假警报）
3. 无需手动设系统级 CUDA_PATH

### 验证（6 项全过）
- gpu.py probe 通过：xp=cupy, cupy_available=True, 无 fallback
- cupy JIT 编译 kernel 成功（astype/abs/std/all）
- 计算在 sm_120 设备执行（cp.cuda.Device(0)）
- monitor._analyze_region cupy 路径：气泡检测 + pair 检测正确
- numpy vs cupy 结果一致
- TUI 显示"已启用 (NVIDIA GeForce RTX 5070)"，警告消失

### 修改文件
- `core/gpu.py`：import cupy 前加 warnings.filterwarnings 过滤假阳性警告
- （环境层面）安装 cupy-cuda12x[ctk] wheels

## AutoReplyer.Kairl — 修复启动硬件加速后无法回复（第二次 CuPy 修复）

### 根因（不同于上一次）
gpu.py 用 `cp.cuda.is_available()` 判断 cupy 可用性，但该方法只检查 GPU 硬件/驱动，不检查 CUDA toolkit headers。本机有 RTX 5070 + nvcuda.dll，is_available() 返回 True，xp 被设为 cupy。但 cupy 实际执行 `astype()` 等 kernel 编译操作时，因缺少 CUDA_PATH/toolkit headers 抛 `RuntimeError: Failed to find CUDA headers`。异常在 monitor._analyze_region 中触发，被 _scan_group 的 except 捕获，导致每次扫描失败、永远检测不到消息、永远不回复。

上一次修复（line 69-86）处理的是"cupy 数组在 Python 控制流中的行为问题"，没覆盖"cupy 运行时编译失败"这个漏洞。

### 修复（根因优先，非补丁）
1. **core/gpu.py**：用实际计算 probe（astype+abs+std）替代 is_available() 做真实可用性检测；cupy 没装时不再 raise ImportError 而是回退 numpy（GPU 加速是可选增强，缺失不应崩溃整个程序）；新增 `gpu_fallback_reason` 记录回退原因
2. **ui/tui.py**：_gpu_status_str 引入 cupy_available 检查，cupy 不可用时显示"已回退 CPU (原因)"，不再误导性地显示"已启用"

### 环境证据
- Python 3.12 + cupy-cuda12x 14.1.1，cuInit 成功，1 个 NVIDIA 设备
- 但 cupy astype 抛 RuntimeError: Failed to find CUDA headers（缺 CUDA_PATH）
- 修复后 xp 回退 numpy，monitor 恢复正常

### 验证
- xp is numpy=True，cupy_available=False，gpu_fallback_reason="CuPy 运行时不可用: RuntimeError"
- _analyze_region numpy 路径：无回复→already_replied=False（能触发），有回复→True（跳过）
- TUI 状态正确显示"已回退 CPU"

- `core/gpu.py`、`ui/tui.py`
